# VeilBrowser AgentOS 端到端工作流推演与前端交互清单

> **背景**: 用户指出，在真正开始后端 MCP 开发之前，需要将整个应用的使用流程（从零配置到执行任务）推演清楚。只有跑通了这套逻辑闭环，我们才会清晰知道前端各个页面还差什么交互弹窗和状态机逻辑。

## 场景：用户要求 Agent 去 Twitter 给特定大V点赞留言

这是一个非常典型的 RPA + LLM 任务。我们将推演用户如何操作我们的前端完成这条流水线。

---

### 第一阶段：基础设施准备 (Infrastructure Setup)

如果底层指纹和网络不安全，Agent 一出门就会被封号。所以用户第一步肯定不是直接聊天。

1. **代理网络配置 (Proxies)**
   - **业务流**: 用户进入“代理网络”，点击“添加节点”，输入一条住宅代理（Residential Proxy）的 Socks5 信息。
   - **交互点**: 点击添加 -> 弹出表单弹窗 (已有) -> 点击保存 -> 后端入库并返回 (已有) -> 点击“测速”验证代理连通性。
   - **前端缺失**: “绑定代理厂商 API” 按钮目前是个空壳，可能需要一个对接 BrightData/IPRoyal 的配置弹窗。

2. **沙盒容器创建 (Contexts)**
   - **业务流**: 用户进入“沙盒容器”，点击“创建隔离容器”。系统通过一套 L4 洗牌算法（基于上述代理节点）生成一套独一无二的硬件指纹和 Canvas 噪声。
   - **交互点**: 容器列表中多出一条 `CTX-XYZ`。卡片上显示当前挂载的 OS、浏览器内核。
   - **前端缺失**:
     - 目前新建容器是前端 mock 生成的，**未让用户选择绑定哪个“代理网络”**。新建容器弹窗应该有“代理节点”下拉框。
     - “容器配置 (Settings)”按钮是空的，点击应该能弹出一个侧边栏抽屉，允许修改它的 UA, WebGL 噪声级别, AudioContext 等极客参数。

---

### 第二阶段：业务资产准备 (Asset Preparation)

环境有了，代理有了，但是账号去哪拿？执行流是什么？

3. **鉴权金库 (Credentials)**
   - **业务流**: 用户进入“鉴权金库”，选择“业务社媒账号” Tab。点击“添加新凭证”。输入 Platform (`twitter.com`)，Username (`@xxx`)，Password (`xxxx`)。
   - **交互点**: 数据以 AES-256 (或 mock) 存入本地库。大模型（LLM）绝对不应该也不能直接读到这个明文密码。
   - **前端缺失**: 鉴权库页面的 UI 已经很完整了，目前交互缺失较少。

4. **技能生态库 (Skills)**
   - **业务流**: 用户进入“技能生态库”，其实就是 Agent 的 SOP (标准作业程序)。用户新建一个叫“Twitter养号”的 Markdown 剧本。
   - **脚本内容**:
     ```markdown
     # 技能：Twitter 点赞

     ## 参数

     - {{target_user}}

     ## 步骤

     1. 打开 https://x.com/{{target_user}}
     2. 检查右上角是否已登录。如果未登录，请求调用 Credentials 库中 twitter.com 的账号信息进行登录。
     3. 找到第一条推文，点击点赞 (Like) 按钮。
     ```
   - **前端缺失**: 编辑器虽然有了，但是 Markdown 到动态执行图 (Dynamic Graph) 的“编译/解析”反馈缺失。用户点击保存后，应该有一个状态指示器显示“AST 解析成功”。

---

### 第三阶段：中枢发号施令 (Command & Orchestrate)

万事俱备，终于轮到真正的 Agent 入场了。

5. **总控大盘 (Dashboard) - AI 交互**
   - **业务流**: 用户在 Dashboard 的聊天框输入：“请用 CTX-XYZ 这个沙盒，执行《Twitter养号》技能，目标大V是 @elonmusk”。
   - **LLM Agent 推理流**:
     1. Agent 解析意图，识别出 `profile_id=CTX-XYZ`。
     2. 识别出所需挂载技能 `skill=twitter_like`。
     3. 识别出动态参数 `target_user=@elonmusk`。
     4. Agent 调用底层向系统注册一个 `TaskRun` 任务流水线。
     5. Backend 发送 `launch_browser` 命令启动这台拥有特定代理、独特指纹的 CEF 容器。
   - **界面响应**: Dashboard 右侧的“Live Browser Frame (实时屏幕流)”立刻由黑屏亮起，开始接收 Base64 传回来的实时截图。
   - **前端缺失**: 聊天区目前对这类任务发射缺乏明确的结构化卡片渲染。如果是发射任务，聊天流应当弹出一个 "Mission Confirmed" 的结构化卡片，而不仅仅是文字回答。

---

### 第四阶段：执行与溯源 (Execution & Telemetry)

6. **任务流水线 (Task Runs)**
   - **业务流**: 任务正在后台跑。用户如果嫌 Dashboard 太拥挤，可以切换到“任务流水线”页面。
   - **界面交互**: 列表显示 `task_xyz (Status: Running)`。右侧是 Terminal 风格的 Telemetry 遥测输出，显示 `[INFO] Injecting mouse jitter... [SUCCESS] Login element fully rendered.`。
   - **前端缺失**:
     - “停止/取消任务” 的红色急停按钮。
     - 以及，如果执行中发生了验证码 (Captcha)，如何将“等待人类接管”的状态反映到 UI 上（比如一个闪烁的红色警告条）？

7. **工件库 (Artifacts)**
   - **业务流**: 任务跑完了。推文点赞成功的截图去哪了？
   - **界面交互**: Agent 自动将“点赞截图.png”推送到工件库。
   - **前端缺失**: 列表目前有了下载和删除，但最好有一个“点击缩略图直接预览图片或 JSON”的弹出层。

---

## 结论与行动纲领

当前的 3-Pillar UI 架构非常漂亮，但如果要让产品“立得住”可以演示，我们需要先在前端填补以下核心交互：

1. **Contexts 页面**: 补齐“创建容器”时的复杂表单弹窗（选择代理、关联指纹、关联操作系统等）。
2. **Dashboard 大盘**: 补齐当输入特定指令后，渲染“任务发射确认卡片 (Mission Card)”的能力，而不是单纯的聊天气泡。
3. **Artifacts 库**: 补齐一个基础的图片/文本模态框预览 (Image/Text Preview Modal)。
4. **Task Runs**: 补齐“中断 (Abort)”和“人类接管 (Takeover)”的状态机 UI。

通过这四项修补，整个应用才会在视觉上形成一条**天衣无缝的闭环**。完成这些前端交互打磨后，后端真正的 MCP 架构就会有了异常清晰的 API 边界契约。
